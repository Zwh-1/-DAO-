# 链上角色与权限（单一事实来源说明）

**HTTP 层**：各接口需要的鉴权类型（`requireAuth`、`requireRole`、`requireAdmin`）见 [角色/后端路由矩阵.md](./角色/后端路由矩阵.md)（修改路由时请同步更新该矩阵）。**JWT 内 `activeRole` 与切换接口**见 [角色/身份切换与 activeRole.md](./角色/身份切换与%20activeRole.md)。

## 实现选型（混合）

- **`PlatformRoleRegistry`**（`contracts/governance/PlatformRoleRegistry.sol`）：基于 OpenZeppelin `AccessControl`，管理 `challenger` / `oracle` / `guardian` / `dao` 等应用角色（`grantAppRole` / `revokeAppRole`）。
- **`ArbitratorPool`**：复用现有 `isArbitrator(address)` 解析 `arbitrator`。
- **未单独上链的角色**：`member` 在链模式下由后端在解析流程中作为 **基线** 赋予任意 SIWE 地址；可后续改为必须在 `PlatformRoleRegistry` 中持有 `ROLE_MEMBER` 才生效。

## ROLES_SOURCE 配置

| 值 | 含义 |
|----|------|
| `memory`（默认） | 角色仅来自后端内存表 `getMemberRoles` / `setMemberRoles`，与链上无关。适用于本地开发、无 RPC。 |
| `chain` | 角色**仅**由 `resolveRolesFromChain(address)` 根据链上合约 view 调用聚合，**不再**合并内存表（除非解析抛错且显式降级策略启用）。 |
| `chain_with_memory_fallback` | 优先链上；若 RPC 失败或合约未配置，回退到内存表，并可在日志中看到警告。 |

环境变量：`ROLES_SOURCE`（见后端 `config.js`）。

## 环境变量（链上读取）

| 变量 | 说明 |
|------|------|
| `RPC_URL` | JSON-RPC 端点，链上角色解析必填（`chain` 模式）。 |
| `PLATFORM_ROLE_REGISTRY_ADDRESS` | `PlatformRoleRegistry` 部署地址；用于 `oracle`、`challenger`、`guardian`、`dao` 等 `AccessControl` 角色。未配置时这些角色在链模式下不会从该合约读出。 |
| `ARBITRATOR_POOL_ADDRESS` | `ArbitratorPool` 地址；用于 `arbitrator`：`isArbitrator(address)`。 |

## 角色与链上判定对照

| 后端 `RoleId` | 链上判定（`ROLES_SOURCE=chain` 时） |
|---------------|-------------------------------------|
| `member` | **任意通过 SIWE 的地址**均赋予（基线成员），不依赖单独链上布尔；若需在链上收紧，可在后续版本改为 `PlatformRoleRegistry` 的 `member` 角色。 |
| `arbitrator` | `ArbitratorPool.isArbitrator(address) == true`（需配置 `ARBITRATOR_POOL_ADDRESS`）。 |
| `challenger` | `PlatformRoleRegistry.hasRole(keccak256(abi.encodePacked("challenger")), address)`。 |
| `oracle` | `PlatformRoleRegistry.hasRole(keccak256(abi.encodePacked("oracle")), address)`。 |
| `guardian` | `PlatformRoleRegistry.hasRole(keccak256(abi.encodePacked("guardian")), address)`。 |
| `dao` | `PlatformRoleRegistry.hasRole(keccak256(abi.encodePacked("dao")), address)`。 |

角色字节与合约中 `keccak256(abi.encodePacked("<roleId>"))` 一致；后端使用 `ethers.id("<roleId>")` 对齐。

## 变更权限的路径

- **生产环境**：禁止通过 `POST /v1/guardian/roles` 直接改内存角色；应通过 **`PlatformRoleRegistry.grantRole` / `revokeRole`**（需 `DEFAULT_ADMIN_ROLE`）或治理执行。
- **开发环境**：仍可调用 Guardian 接口便于联调（见 `guardian.routes.js`）。

## JWT 与刷新

- 登录：`POST /v1/auth/verify` 在签发 JWT 前按 `ROLES_SOURCE` 解析 `roles`。
- 刷新：`POST /v1/auth/refresh-roles`（需 `Authorization: Bearer <token>`），按当前链上状态重新签发 JWT，避免链上角色变更后长期持有旧 `roles`。

## 回滚

将 `ROLES_SOURCE=memory` 即可恢复纯内存行为（仅建议非生产或紧急排障）。
