# TrustAid 权限模型（架构决策）

**版本**：1.0  
**范围**：HTTP（JWT / Admin Token）、与链上 `PlatformRoleRegistry` 的关系；不包含合约 Multisig/Timelock 部署细节。

---

## 1. 应用角色权威源（`ROLES_SOURCE`）

| 取值 | 含义 | 适用场景 |
|------|------|----------|
| `memory`（默认） | JWT 内角色主要来自内存成员表（`getMemberRoles`），登录时可不读链 | 本地开发、无 RPC |
| `chain` | 角色**完全**以链上 `PlatformRoleRegistry` + `ArbitratorPool.isArbitrator` 为准；解析失败时登录返回 **503** | 生产强一致 |
| `chain_with_memory_fallback` | 链上角色与内存角色**合并去重**；链失败时仍可使用内存角色 | 预发布、链不稳定时 |

**约定**：链上 grant/revoke 角色后，客户端应调用 **`POST /v1/auth/refresh-roles`** 刷新 JWT（见 [`server.js`](../../backend/src/server.js)）。

---

## 2. 守护者模块 HTTP（`/v1/guardian/*`）

| 机制 | 说明 |
|------|------|
| **当前实现** | 全部使用 **`requireAdmin`**：请求头携带 `ADMIN_TOKEN`（`Authorization: Bearer` 或 `x-admin-token`），**与 JWT 中的 `guardian` 角色无关**。 |
| **产品语义** | 「紧急熔断 / 黑名单」属于**运营或协议管理员**能力，使用独立高熵密钥，避免与终端用户 JWT 混用。 |
| **可选增强（路线图）** | 在 Admin 校验通过后，再 **`hasRole(GUARDIAN)`** 读链核对，形成双因子；未实现前以本文档为准。 |

---

## 3. 治理内存 API（`/v1/governance/propose`、`/vote`）

| 决策 | 说明 |
|------|------|
| **角色要求** | `POST /propose`、`POST /vote` 需 **`requireRole("dao")`**，与前端 `RoleGuard required="dao"` 及 `RoleId.dao` 对齐。 |
| **身份字段** | 提案人、投票人地址取自 **`req.auth.address`**（SIWE 钱包），禁止依赖未赋值的 `req.user`。 |

公开 **`GET /proposals`** 保持无需登录（演示与监控）。

---

## 4. 支付通道参与者校验

当 **`NODE_ENV=production`** 或设置 **`FORCE_CHANNEL_PARTICIPANT_CHECK=1`** 时：

- **开通通道**：`participant1` / `participant2` 之一须为当前 JWT 地址。
- **变更状态 / 退出 / 提款 / 关闭**：调用方须为该通道的 **participant1 或 participant2**。

开发环境默认不强制，便于无钱包联调；生产必须通过上述规则。

---

## 5. 前端 `RoleGuard`

仅基于本地会话 **`roles`** 控制**页面可见性**，**不能替代**后端鉴权；任何敏感操作以 HTTP 返回码为准。

详见 [`frontend/features/governance/RoleGuard.tsx`](../../frontend/features/governance/RoleGuard.tsx) 文件头注释。

---

## 6. 管理员 Token 存储（前端）

当前守护者调试页使用 **`localStorage.adminToken`**。中长期建议改为 **httpOnly Cookie** 或 **短时代理接口**，降低 XSS 窃取面；实现前本文档仅作风险披露。
